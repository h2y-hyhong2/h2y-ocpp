## 1. 아키텍처 의사결정 (Architecture Comparison)
대용량 트래픽 처리와 ~~Phase 2 수평 확장~~ 비용을 고려하여 두 가지 아키텍처 후보를 비교 분석하고 최종 안을 채택하였습니다. (Phase 2 수평 확장은 개발 제외)

* **제 1안: Spring Integration EIP 기반 아키텍처 (채택):**
  * **핵심:** 게이트웨이와 비즈니스 코어 간 통신을 Spring Integration 메시지 채널(`ocppRawChannel`)로 추상화하고 Kafka를 메시지 버스로 연동.
  * **선정 사유:** 로컬 단독 구동(Phase 1)과 ~~HA 클러스터 분산(Phase 2)~~ 간에 소스코드 수정이 전혀 필요 없고, 인프라 설정 교체만으로 스위칭이 가능하며 비즈니스 도메인 코드가 외부 미들웨어 인터페이스에 오염되지 않음. (Phase 2 분산은 개발 제외)
* **제 2안: Spring Boot + Spring Kafka 수동 구현 아키텍처 (유보):**
  * **핵심:** 개발자가 직접 서비스 클래스 내에 `@KafkaListener` 및 `KafkaTemplate`을 사용하여 수동 비동기 처리.
  * **유보 사유:** 단독 구동에서 분산 구조로 전환 시 게이트웨이 및 세션 라우팅을 위한 Redis Pub/Sub 코드를 대거 수동으로 재구축해야 하여 이중 개발 공수 발생.

## 2. Spring Integration 기반 메시지 파이프라인 설계
충전기로부터 수입되는 업링크(Uplink) 및 다운링크(Downlink) 패킷의 물리/논리 분할 파이프라인 구조입니다.

```
[충전기 (CP)] 
    │ (WebSocket 연결)
    ▼
[WebSocket Inbound Adapter]
    │ (Raw JSON 패킷 추출)
    ▼
[ocppRawChannel (KafkaChannel)]  <───> [Kafka Broker (메시지 완충 및 장애격리)]
    │ (비동기 Consume)
    ▼
[OCPP(Open Charge Point Protocol) JSON-to-DTO Transformer]
    │ (DTO 객체 변환)
    ▼
[ocppDtoChannel (DirectChannel)]
    │
    ▼
[Payload-Type Router]
    ├── (BootNotification/Authorize) ──> [OCPP(Open Charge Point Protocol) 프로토콜 핸들러 (Postgres 저장)]
    ├── (MeterValues) ─────────────────> [ClickHouse 벌크 적재 라이터]
    └── (StatusNotification) ──────────> [장애 전이 감지기 (웹소켓 알림 전송)]
```

## 3. 하이브리드 세션 관리 설계
인프라 구성 환경에 따라 유연하게 교체할 수 있도록 `SessionRepository`를 추상화 설계합니다.

* **Local Session Store (Phase 1):**
  * 단일 게이트웨이 서버 환경에 적합하도록 JVM 내부 `ConcurrentHashMap`을 이용하여 충전기 소켓 세션을 관리. 외부 캐시 서버 연결 오버헤드를 줄여 저사양 싱글 서버에 최적화.
* ~~**Distributed Session Store (Phase 2):**
  * L4/L7 로드밸런서 배치에 따라 여러 웹소켓 게이트웨이 노드로 세션이 흩어지는 구조에 대응.
  * Redis `Active Connection Map`에 충전기별 접속 노드 정보를 동적 보관하고, Redis Pub/Sub 채널을 이용하여 대상 충전기가 물려 있는 게이트웨이 노드로 제어 명령을 원격 브로드캐스팅 라우팅 처리.~~ (개발 제외)

## 4. CQRS 기반 하이브리드 데이터베이스 설계
트래픽의 성격에 따라 데이터베이스 아키텍처를 물리적으로 분리(CQRS) 설계하여 고속 삽입 및 실시간 쿼리 성능을 모두 확보합니다.

* **OLTP (PostgreSQL):**
  * **역할:** 정적 자산 데이터(충전소/기기 정보), 회원 마스터, 요율표, 금융/세무 정합성이 보장되어야 하는 최종 과금 CDR(Charge Detail Record) 정산 원장(`charging_transaction`) 저장.
  * **특징:** 트랜잭션 안전성(ACID) 보장 및 일 단위 테이블 파티셔닝 적용.
* **OLAP (ClickHouse):**
  * **역할:** 초 단위로 인입되는 실시간 계측 데이터(`meter_value_history`), 통신 원본 JSON 패킷 로그(`ocpp_raw_log`) 저장.
  * **특징:** Columnar Storage 지향의 **MergeTree** 데이터베이스 엔진 채택. 10배 이상의 고압축 저장으로 디스크 용량 절감 및 통계 쿼리 연산 속도 극대화.
