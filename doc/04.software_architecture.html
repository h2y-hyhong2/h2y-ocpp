1. 아키텍처 핵심 설계 원칙 (Architectural Principles)

본 시스템은 대용량 비동기 충전기 트래픽의 병목 현상을 해결하고, 폐쇄망 실환경의 제한된 장비 리소스를 극대화하기 위해 다음의 3대 소프트웨어 아키텍처 원칙을 고수합니다.

📡 비차단 비동기 파이프라인 (Loose Coupling)
충전기 웹소켓 커넥션을 받아 패킷을 수집하는 WebSocket Gateway와 정산 연산 및 DB 적재를 담당하는 Business Core 사이에 Apache Kafka를 완충지대로 배치합니다. 백엔드의 DB 병목이나 PG사 결제 연동 지연 등이 실시간 소켓 연결 유지에 지장을 주지 않습니다.
💾 하이브리드 세션 관리 (Stateful Agent + Stateless Service)
충전기 세션별로 상태 정보만을 지니는 경량화된 OcppSessionAgent (Stateful) 객체를 JVM 메모리에 보존하여 상태 검증을 초고속으로 수행하고, 무거운 정산 계산이나 CRUD 트랜잭션 처리는 Spring의 Stateless Service 싱글톤 빈들이 가상 스레드 상에서 처리하도록 이원화합니다.
🗄️ 데이터 저장소 CQRS (Command Query Responsibility Segregation)
회원 원장, 충전기 정보, 그리고 무결성이 요구되는 최종 매출 정산서(CDR)는 PostgreSQL(OLTP)에 엄격히 보관하고, 15초 주기로 밀려드는 무수한 계측 데이터(MeterValues)와 원본 JSON 송수신 패킷은 대량 압축 쓰기에 강한 ClickHouse(OLAP)에 시계열 분리 적재하여 디스크 병목을 해결합니다.

2. 소프트웨어 다층 레이어드 구조 (Software Layered Architecture)

전체 아키텍처 모듈 레이어는 결합도를 차단하고 유연한 이기종 프로토콜 확장을 지원하기 위해 다음과 같이 계층화 설계되었습니다.

2.1. 각 소프트웨어 레이어 설명

  • 웹소켓 게이트웨이 핸들러: 소켓 연결 및 수명 주기를 제어하며, 유입되는 OCPP JSON의 정합성 스키마를 1차 필터링하고 로컬 ConcurrentHashMap에 세션을 매핑하여 상시 유지합니다.
  • Kafka 이벤트 파이프라인: 유입되는 모든 실시간 패킷은 raw-events 큐로 전달되어 순서대로 처리되고, 원격 제어 명령은 control-commands 토픽을 통해 분배됩니다.
  • 가상 스레드(Virtual Threads) 처리 엔진: Java 25 가상 스레드를 적극 할당하여, 대량의 트랜잭션 및 DB 쓰기 차단 작업이 발생하더라도 가용 시스템 스레드를 블로킹하지 않고 고속 병렬 처리합니다.

3. 웹소켓 세션 관리 및 세션 객체 상세 설계 (Session Design)

3.1. `OcppSessionAgent` 데이터 스펙

동시성 연결 증가 시 JVM의 힙(Heap) 공간 압박을 해소하기 위해, 동작 제어 로직을 일절 베재하고 핵심 상태 변수만을 보존하도록 단순 가볍게 설계했습니다.

public class OcppSessionAgent {
    private final String chargePointId;       // 충전기 고유 식별 시리얼 (Key)
    private final WebSocketSession session;    // JVM 실제 물리 웹소켓 소켓 연결 객체 참조
    private String ocppVersion = "1.6J";       // 프로토콜 버전 (1.6J 또는 2.0.1)
    private SessionState state = SessionState.CONNECTED; // 세션 수명 주기 상태
    private String activeTransactionId = null; // 현재 진행 중인 충전 트랜잭션 식별 ID (미충전 시 null)
    private Instant lastHeartbeatTime;        // 최종 하트비트 패킷 수신 시각 (커넥션 타임아웃 감지용)
    
    // Getters & Setters
}

3.2. 세션 수명 주기 상태 전이 모델

각 세션 객체는 기기 통신 단계에 따라 다음과 같이 일관되게 트래킹되며, 최종 소켓 단절 시 즉시 클린업 맵에 처리됩니다.

4. [Phase 2] 다중 노드 분산 환경에서의 원격 명령 라우팅 흐름 (Routing Flow)

로드밸런서 하부 다중 게이트웨이 노드 구성 시, 어드민 제어 명령이 충전기가 속하지 않은 다른 서버 노드로 인입되더라도 Redis Pub/Sub 채널을 이용하여 연결 대상 소켓이 존재하는 노드로 원격 제어 명령 패킷을 정확히 유도 및 전송합니다.

⚠️ 분산 상태 동기화 및 메시지 유실 예방 정책
  • 노드 맵 등록: 충전기 소켓 세션 연결 수립 시 게이트웨이 노드는 즉시 Redis Hash 테이블인 active_session_map에 [충전기ID] : [노드ID] 데이터를 기록합니다. 소켓 해제 시 해당 데이터를 자가 삭제(Clean) 처리합니다.
  • 장애 극복 라우팅: 대상 노드가 순간적인 크래시(Crash)로 인해 Redis Pub/Sub을 받지 못할 경우, 즉각 에러 처리를 하지 않고 Apache Kafka control-commands-retry 토픽으로 제어 명령을 재우회 전송하여 3회 재시도를 보장합니다.

5. EAI 관점의 통합 아키텍처 매핑 (EAI Integration Architecture Map)

솔루션의 통합 구성 요소를 고전적 EAI(Enterprise Application Integration) 엔터프라이즈 통합 유형과 비교 매핑하여 시스템 결합도를 도식화한 내용입니다.

📌 EAI 설계 구성 요소 분석
  • Source Adapters (송신 어댑터):
    • WebSocket Gateway: OCPP 프로토콜을 전송 매체(TCP/IP 소켓)로부터 발췌하는 입력 어댑터.
    • REST API Adapter: 내부 어드민 및 외부 파트너 시스템의 HTTP 트랜잭션 수신 어댑터.
  • EAI Message Bus (메시지 통신 채널 - Kafka): 중앙 메시지 백본으로서 비동기 큐잉(Queueing)과 브로드캐스트 분배를 도맡으며 메시지 순서 및 백로그 자가 제어 담당.
  • EAI Router & Transformer (통합 변환 엔진 - Spring Core):
    • Translation & Validation (파싱 & 검증): 다양한 버전(OCPP 1.6 / 2.0.1)의 이기종 프로토콜 JSON을 플랫폼 내부 표준 자바 DTO(Canonical Data Model)로 공통 변환 및 유효성 검증.
    • Enrichment & Calculation (정산 & 보강): 충전기 원천 데이터에 한전 요율 정보 및 충전소 자산 메타 정보를 결합하여 충전 금액 연산 및 프로파일 추가.
    • Routing & Distribution (분배): 가공된 데이터를 목적에 맞게 실시간 트래픽, 정산 원장, 외부 통신 어댑터 등으로 분류하여 실시간 유도.
  • Target Adapters (수신/적재 어댑터): DB 라이터 모듈(JPA/ClickHouse Bulk Writer)과 외부 시스템 프록시 클라이언트(Payment Gateway, Roaming API Client)들이 해당 데이터 구조를 최종 적재 또는 대외 전송하는 수신 어댑터(Target Adapters) 역할을 성실히 대행합니다.
