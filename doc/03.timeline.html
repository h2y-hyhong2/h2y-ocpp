1. 프로젝트 마일스톤 (Project Milestones)

2.1. [1단계] 개발 환경 및 인프라 구축 (Week 1 - 2)

단일 가상 머신(VM) 및 호스트 컴퓨터 내에 백엔드 구동 환경과 하이브리드 데이터베이스 시스템을 안착시키는 단계입니다.

  • PostgreSQL 및 ClickHouse 초기 설정: 관계형 데이터 모델링(PostgreSQL) 및 대용량 통신 로깅을 위한 MergeTree 스키마(ClickHouse) 생성.
  • 메시지 큐 기동: 내장형 단일 Kafka Broker(KRaft Mode) 설정 적용 및 하루 단위 Retention 정책 설정.
  • 스프링 부트 빌드 뼈대 구성: Java 25 가상 스레드(Virtual Threads) 처리 기능을 백엔드 코어에 활성화.
  • CI/CD 환경 설정: 개발망 및 내부 패키징 환경 자동 테스트 파이프라인 연동.

2.2. [2단계] Phase 1 백엔드 개발 - OCPP 1.6J 구현 (Week 3 - 5)

웹소켓 통신 인프라 및 충전기와 서버 간의 단일 노드 양방향 메시징 흐름을 구축합니다.

  • WebSocket Gateway 모듈: 충전기와의 소켓 통신 접점을 열고, 메모리 상에 세션 맵(LocalSessionStore)을 관리하도록 설계.
  • Kafka 파이프라인 연계: 수신 메시지는 raw-events 큐로 발행하고, 명령 발송은 control-commands 큐를 통해 게이트웨이에 라우팅.
  • OCPP 1.6J Core 기능 구현: 기기 연동에 필수적인 Core 프로파일 메시지 7종(BootNotification, Heartbeat, StatusNotification, Authorize, StartTransaction, StopTransaction, MeterValues) 개발 완료.
  • ClickHouse 미터 수집 최적화: 15초 주기의 미터 정보 벌크 쓰기(Bulk Write) 배치 로직 설계.

2.3. [3단계] Phase 1 프론트엔드 개발 - 관제 UI 및 3D 시각화 (Week 6 - 7)

사용자 편의성과 고화질 엔터프라이즈 모니터링 대시보드 화면을 Vue 3 및 WebGL 기술을 활용하여 제작합니다.

  • 어드민 콘솔 웹 개발: Vue 3 (Composition API) 및 Pinia 기반 대시보드 템플릿 설계.
  • 3D 디지털 트윈 관제: Three.js 기반 충전소 건물 렌더링 및 충전기 동작 상태 아우라(Glow) 효과 시각화.
  • 실시간 패킷 로그 뷰어: OCPP 원본 데이터(JSON)를 실시간으로 모니터링하고 일시정지, 필터링, 정밀 보기 기능 적용.
  • 오프라인 지도 대응: 외부망이 끊긴 환경에서도 GeoJSON 데이터를 활용해 Leaflet.js 오프라인 지도 렌더링 지원.

2.4. [4단계] 정산 엔진, FOTA 및 Phase 1 통합 검증 (Week 8 - 9)

정산 로직 구현과 단일 기동 환경 최적화를 진행하며 마일스톤 1(M1) 단계를 확정 짓습니다.

  • TOU (Time-of-Use) 정산 엔진: 계절별, 시간대별 및 회원 등급별 할인 룰이 적용된 사용 요금 실시간 정산 로직 탑재.
  • 정산 무결성 확보: 회계 일치성을 위해 충전 상세 기록(CDR) 최종 내역을 PostgreSQL 트랜잭션 데이터베이스에 격리 저장.
  • 로컬 FOTA 서버: 폐쇄망 인접 저장소를 활용한 안전한 펌웨어 패키지 배포 시스템 연동.
  • 단일 호스트 성능 튜닝: 내장 Kafka JVM 힙메모리(512MB) 강제 제약 및 ClickHouse 단일 쿼리 최대 자원 한계 제약(4GB).
  • 통합 검증: 충전기 시뮬레이터를 사용해 2,000대 연결 상황에서 통신 끊김 및 장애 복구 로직 통합 QA.

2.5. [5단계] Phase 2 스케일 아웃 및 OCPP 2.0.1 고도화 (Week 10 - 11)

트래픽 분산과 스케일 아웃에 유연하게 대응하고 OCPP 최신 표준 규격을 활성화하는 고도화 과정입니다.

  • 로드밸런싱 및 Redis 세션 공유: L4/L7 로드밸런서 배치 및 서버 다중화 시 세션 실시간 공유를 위한 Redis Active Connection Map 연동.
  • 제어 명령 Pub/Sub 라우팅: 코어 서버가 Redis Pub/Sub을 활용하여 대상 충전기가 연결된 특정 게이트웨이 노드로 원격 제어 명령을 실시간 전송.
  • 데이터베이스 다중화: PostgreSQL 복제본(Primary-Standby) 및 ClickHouse 분산 레플리카 환경 구성.
  • OCPP 2.0.1 규격 반영: 2.0.1 전용 메시지 스키마 변환 어댑터 탑재 및 Smart Charging, 신규 트랜잭션 규격 개발.

2.6. [6단계] 대용량 부하 테스트 및 최종 배포 (Week 12)

실제 폐쇄망 배포본을 구성하고 극한의 트래픽(2,000 TPS) 상황에서 백엔드 및 인프라의 안정성을 최종 점검합니다.

  • 2,000 TPS 부하 테스트: 가상 부하 제너레이터 구동 및 메시지 유입 병목(임계 Latency 평균 100ms) 여부 최종 검증.
  • Zero-Dependency 패키징: 폐쇄망 환경을 위한 프론트엔드 Vue 빌드 파일의 Spring Boot 리소스 static 내장 포장 및 단일 JAR 빌드.
  • 컨테이너 배포 세트 마련: 배포 자동화를 위한 Dockerfile 최적화 및 Kubernetes Helm Chart 패키징.
  • 인수 및 교육 지원: 릴리즈 노트 공유 및 시스템 설치/운영 교육 자료 인계.

4. 리스크 관리 계획 (Risk Management)

⚠️ 디스크 및 I/O 병목 완화 (ClickHouse Bulk Write)
미터값 등 초 단위 실시간 유입 시 디스크 쓰기 병목이 예상됩니다. 런타임 상에서 가상 스레드 기반 인메모리 버퍼에 메시지를 임시 적재한 후, 1초 단위 또는 1,000건 단위로 ClickHouse에 모아서 Insert를 수행하도록 벌크 처리 로직을 필수 설계합니다.
🗺️ 오프라인 망 대응 (Map Data local serving)
군부대 등 완전 격리된 폐쇄망 환경에서는 외부 지도 API를 호출할 수 없습니다. 사전에 해당 부지 및 지역의 OSM(OpenStreetMap) GeoJSON 벡터 맵 소스를 static 파일로 서버 스토리지에 탑재하여 오프라인에서 Leaflet.js가 지도 타일을 완전 로컬 서빙하도록 처리합니다.
⚡ 분산 노드 간 커넥션 세션 갱신 딜레이
다중 서버 스케일 아웃 상태에서 급격한 연결 끊김 발생 시 Redis 세션 맵과의 동기화 지연이 발생할 수 있습니다. 제어 명령 발송 시 대상 노드가 세션 만료로 인식되면 즉시 에러 처리를 하지 않고, Kafka 재처리 토픽(control-commands-retry)을 이용해 3회 이내의 재시도 로직을 보장합니다.
